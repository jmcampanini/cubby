Cubby Milestone 6

Status, Doctor, and Prune

A human-readable walkthrough of what changes, what matters, and where the implementation can get tricky.

Use ← / →, Space, Home, End, or the buttons below.

The story

Milestone 6 makes Cubby explain itself.

Earlier milestones create and remove profile symlinks. This one adds observability and cleanup.

Status

What is linked? From which source? Which profile? Is anything drifting?

Doctor

Is this setup healthy enough to trust? If not, what exactly is wrong?

Prune

Can Cubby safely remove stale links whose source files are gone?

Before milestone 6

What you can already do

  • Register one or more sources in host .cubby.toml.
  • List sources and profiles with source list and profile list.
  • Link/unlink selected profiles from host defaults, CUBBY_PROFILE, or --profile.
  • Avoid unsafe overwrites through conflict detection, conflict skipping, and dry-run previews.
  • Protect the public host repo with gitignore check and gitignore sync.
Before milestone 6

What is still hard today

No inventory of real state

You can link files, but there is no first-class command to ask: “what links are currently in my host repo?”

No stale-link cleanup

If a source file is deleted, the host symlink can dangle until you find and remove it manually.

No aggregate health check

Gitignore, conflicts, missing sources, and dangling links are separate concerns. There is no single “is this OK?” command.

No drift vocabulary

Links can exist but no longer match the current config, ignore rules, or source profile declarations.

After milestone 6

What you should expect to do

$ cubby status LINKED nvim/init.work.lua [source=work profile=work target=nvim/init.work.lua] DRIFT old.work [source=work profile=work target=old.work reason=dangling] $ cubby doctor MISSING_GITIGNORE *.work.* DANGLING old.work [source=work] CONFLICT nvim/init.work.lua host path already exists $ cubby prune REMOVE old.work

Exact wording may differ, but the important property is stable, scriptable, line-oriented output.

Mental model

No active-profile state file

Cubby does not remember what profiles are “active.” It infers current state by walking the host repo.

1

Walk the host repo.

2

Find symlinks.

3

Resolve each symlink target.

4

If the target points inside a registered source, classify it as Cubby-managed.

5

Infer source, profile, target path, and drift from current config and filesystem state.

What matters most

The core classification must be trustworthy.

  • Managed vs unmanaged: only symlinks pointing into registered sources are Cubby's responsibility.
  • Dangling but still recognizable: stale links into a source should still be prunable even when the target no longer exists.
  • Source ownership: target path maps to the correct registered source.
  • Profile ownership: profile comes from the existing filename grammar and that source's declarations.
  • Drift reasons: problems should be explicit, not hidden behind a generic “bad link.”
Implementation structure

Primary mechanisms being proposed

1. Diagnostic project loader

Strict loading stays for normal commands; doctor gets a loader that can collect source issues without stopping at the first one.

2. Host symlink scanner

Walks the host repo, skips .git, classifies symlinks, and returns stable managed-link records.

3. Drift classifier

Compares existing links to current source config, profile grammar, ignore rules, target existence, and expected path mapping.

4. Doctor aggregator

Combines gitignore checks, source issues, missing requested profiles, dangling links, drift, and link conflicts.

Strict vs diagnostic loading

This is a key design seam.

Strict loader

Used by commands that need a valid project before doing anything: link, unlink, status, prune, gitignore.

Diagnostic loader

Used by doctor so it can say: “source A is missing, source B has missing gitignore patterns, and source C has conflicts.”

Human guidance: care that this seam exists. Do not care too much about the exact function names as long as the behavior is separated.

Managed link record

The scanner should produce facts, not print output.

type ManagedLink struct { HostRelPath string RawTarget string TargetPath string SourceName string SourceRelPath string Profile string TargetExists bool DriftReasons []string }

The command layer can then render these facts as status, use them for prune, or aggregate them in doctor.

Status

Think of status as read-only inventory.

  • It loads a valid project.
  • It discovers managed host symlinks.
  • It reports source, profile, target, and drift.
  • It should exit zero if it successfully reports state, even if the state contains drift.

Human focus: the output must answer “what is this link and should I trust it?”

Prune

Think of prune as the smallest safe mutation.

Remove

Only dangling symlinks whose target path is inside a registered source.

Never remove

Valid managed links, unmanaged symlinks, regular files, directories, or parent directories.

Human focus: pruning safety matters more than pruning convenience.

Doctor

Think of doctor as an aggregate health report.

  • MISSING_SOURCE: a registered source path/config is broken.
  • MISSING_GITIGNORE: required public-safety patterns are absent.
  • MISSING_PROFILE: host default profile is declared by no valid source.
  • DANGLING: managed symlink target no longer exists.
  • DRIFT: managed link no longer agrees with current source/config rules.
  • CONFLICT: a future link for requested profiles would hit a conflict.

Exit code: zero only when no issues are found.

What to care about

Review priorities as a human

  1. Safety boundaries: commands must not mutate unrelated host files.
  2. Reusable classification: status, prune, and doctor should share the same managed-link discovery logic.
  3. Stable diagnostics: predictable prefixes and ordering are better than fancy formatting.
  4. No hidden state: all answers come from config + filesystem.
  5. Failure timing: mutating commands should fail before mutation if the project cannot be trusted.
What not to over-focus on

Details that matter less initially

  • Exact visual styling, as long as output remains scriptable.
  • Whether the scanner package is named hostlinks, state, or something else.
  • Perfectly rich prose in diagnostics. Stable issue type + path + reason is enough.
  • Interactive remediation. Milestone 6 reports and prunes; it does not need a wizard.
  • Cleaning empty parent directories after prune. The plan explicitly avoids that.
Trouble areas

Watch these after the code is implemented

Dangling symlink resolution

EvalSymlinks fails for missing targets. The scanner still needs the cleaned absolute lexical target.

Inside-root checks

Use safe relative-path checks. String prefix checks can confuse /src with /src-old.

Overlapping source roots

If one source root is inside another, pick the most-specific matching source.

Source roots inside host

The host walk should not recurse into registered source roots and classify source-internal symlinks as host state.

More trouble areas

Subtle correctness checks

  • Ignore rules: status/doctor drift must reuse the same ignore semantics as profile discovery.
  • Profile grammar: keep using exact dot-segment matching; do not introduce fuzzy profile matching.
  • Path normalization: filesystem paths use OS separators; ignore patterns and output often want slash-normalized paths.
  • Deterministic ordering: tests and scripts should not depend on walk-order accidents.
  • Case sensitivity: doctor conflict checks should respect the existing host case_sensitive behavior.
  • No mutation in doctor/status: only prune should remove files.
Doctor conflict checks

Reuse the link planner; do not reimplement conflicts.

The plan proposes using existing profile discovery plus linkops.PlanLink for conflicts.

1

Take host top-level requested profiles, if any.

2

Discover source files for valid selected profiles.

3

Run the non-mutating link planner.

4

Report planned fatal conflicts as doctor issues.

Human focus: conflict behavior should stay identical between link --dry-run and doctor.

Testing shape

The tests should prove behavior vertically.

Unit tests

Scanner classification, source ownership, profile inference, dangling recognition, ignore-rule drift, path mismatch.

End-to-end tests

Real host/source repos, real symlinks, status output, doctor exit codes, and prune non-destructive cleanup.

Best signal: one test should prove unmanaged symlinks survive prune.

How to review the final code

A practical checklist

  • Can I explain the scanner's classification rules without reading command code?
  • Do status, doctor, and prune share that scanner?
  • Can doctor report more than one problem at a time?
  • Can a dangling managed link still be identified as managed?
  • Does prune have a tiny mutation surface?
  • Are diagnostics stable enough for tests and shell scripts?
  • Do existing milestones still pass unchanged?
Bottom line

The important idea

Milestone 6 is less about adding three commands and more about adding a reliable model of current host state.

Once that model is trustworthy, status is rendering, prune is one safe mutation, and doctor is aggregation.

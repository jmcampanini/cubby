Your job: do not ask “does this sound good?” Ask “what evidence would prove this works?”

The stated outcome

What must be true after M3?

Behavioral outcome

The existing single-source link/unlink flow becomes realistic enough for daily use:

  • profiles can come from host defaults, env, or flags
  • source repos declare what they provide
  • ignore rules prevent unwanted source files from linking
  • profile discovery is scriptable via profile list

Review question

Can a human run commands and know whether the outcome happened?

If the plan maps each promise to tests, diagnostics, and observable filesystem state, it is reviewable.

Mental model

Three different “profile” concepts

1. Host defaults

# host/.cubby.toml
profiles = ["work", "personal"]

What the host applies when the command provides no explicit selection.

2. Invocation selection

CUBBY_PROFILE=work cubby link
cubby link --profile personal

Per-run override of host defaults.

3. Source availability

# source/cubby.toml
profiles = ["work", "client-acme"]

What a source repo actually provides.

Gotcha: many bugs come from mixing these up. Host defaults choose; source profiles authorize availability.

Config-loader integration

What implementation should smell like

Expected code shape

type HostConfig struct {
    Profiles []string `toml:"profiles" config:"profile" help:"profile to apply; repeatable or comma-separated"`
    Sources  []HostSource `toml:"source"`
}

envLoader, _  := configloader.NewEnvironmentLoader[HostConfig]("cubby", configloader.OSEnv())
flagLoader, _ := pflagloader.NewLoader[HostConfig](cmd.Flags())

host, _, err := configloader.Load(defaults, fileLoader, envLoader, flagLoader)

Review checks

  • --profile is registered before Cobra parses args.
  • Only profile-scoped commands use env/flag overlays.
  • Source configs still load from source repos after host paths resolve.
  • CSV/repeated flag behavior comes from config-loader, not duplicate custom parsing.

Red flag: if link/unlink still call a helper that only reads cmd.Flags().GetStringSlice("profile"), host defaults and env fallback are probably not implemented correctly.

Ignore rules

What “ignore” means in source discovery

Source config

profiles = ["work"]
ignore = [
  "nvim/skip.work.lua",
  "*.draft.*",
  "**/*.secret.*",
]

Rules to check

  • Match against source-relative paths.
  • Normalize separators to /.
  • No slash means basename anywhere.
  • ** means recursive directories.
  • Invalid glob patterns are hard errors.
PatternShould ignoreShould not ignore
nvim/skip.work.luanvim/skip.work.luaother/skip.work.lua
*.draft.*foo.draft.work.md, nested/foo.draft.work.mdfoo.work.md
init.work.luainit.work.lua, nvim/init.work.luanvim/init.personal.lua
**/*.secret.*a/b/foo.secret.workfoo.work
Ignore implementation gotcha

** is not automatic everywhere

Go's standard path.Match / filepath.Match do not provide full gitignore-style recursive ** semantics by themselves.

What to look for

  • A small matcher with explicit tests for **/*.draft.*.
  • Clear invalid-pattern detection.
  • No accidental OS-specific separator behavior.

Failure mode

If ** is treated like plain *, ignored nested files may still link. That violates the milestone outcome even if simple ignores pass.

The plan deliberately keeps gitignore broader than current selection. This is safer because public host repos should not accidentally commit any profile-scoped file that a registered source can provide.

Red flag: if gitignore code starts reading effective command profiles, it may miss patterns for source-declared profiles that are not active today.

Review risks

Potential weak spots in the plan

Unlink env/flag coverage: the plan only requires one e2e for unlink default selection. That is probably fine if link/unlink share a tested selection helper, but verify they actually share it.

Empty flag value: the decision says --profile "" should error and not fall back. Make sure a test captures this edge case, even if it is unit-level.

Strict old schema: old host configs are intentionally outside M3. If backwards compatibility matters, the plan does not cover it.

Multi-source collisions: not part of M3. Avoid adding collision promises here; that is Milestone 4/5 territory.

Implementation red flags

Stop and re-check if you see these

Selection

  • Flag parsing duplicated in multiple commands.
  • Env read directly with os.Getenv while config-loader is also present.
  • Unknown profile checked after linking begins.

Discovery

  • Ignores checked against absolute paths.
  • Backslash path separators leak into matching.
  • .work starts matching as a valid profile file.

Output

  • profile list prints tables by default.
  • Gitignore uses host defaults instead of all source profiles.
  • Tests assert only exit codes, not filesystem state.
Reviewer worksheet

Before approving M3, answer these

  1. Can I explain the difference between host defaults and source availability?
  2. Can I predict effective profiles for host/env/flag combinations?
  3. Is there a test proving unknown selected profiles create no links?
  4. Is ignore matching tested for exact path, basename, glob, recursive glob, and invalid glob?
  5. Does profile list print only source-declared available profiles?
  6. Does gitignore still cover every source-declared profile, even inactive ones?
  7. Do existing M1/M2 tests still pass after the host schema change?

If the answer to all seven is yes, the plan has a high chance of accomplishing its stated outcome and being objectively verifiable.

Go / no-go rubric

What “good enough” looks like

Go

  • Every decision has a corresponding test.
  • Errors happen before side effects.
  • Output remains scriptable.
  • Safety checks stay conservative.

No-go

  • Selection precedence is ambiguous or duplicated.
  • Unknown profiles silently no-op.
  • Ignore semantics are underspecified in implementation.
  • Gitignore narrows to active profiles.

The plan is strong if implementation can be reviewed as a chain:
config → selected profiles → validated profiles → discovered files → filesystem/output assertions.

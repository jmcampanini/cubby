Cubby · Milestone 5

Multi-Source Inventory Slice

This milestone turns Cubby from “it can work with multiple sources” into “we have proven, observable multi-source behavior, with human-friendly tables and JSON for scripts.”

Before vs after

What changes for the user?

Before Milestone 5After Milestone 5
Multi-source behavior exists in pieces, but the inventory command is still placeholder-ish. cubby source list renders a Lipgloss table, and --json emits scriptable inventory.
Source names may be weakly defined. Every source must have a non-empty, unique name using only letters, digits, dashes, and underscores.
A selected profile absent from one source silently produces no work for that source. Cubby explains that skip on stderr, while still doing work in sources that provide the profile.
Existing tests imply multi-source support. End-to-end tests prove multi-source link, unlink, source list, profile list, and gitignore union behavior.

Feature 1

Source inventory becomes real

cubby source list renders a Lipgloss table for humans:

NAME  PATH                         PROFILES
one   /Users/me/dotfiles-work      work,personal
two   /Users/me/dotfiles-client    client-acme

For scripts, cubby source list --json skips styling and prints JSON:

[{"name":"one","path":"/Users/me/dotfiles-work","profiles":["work","personal"]},
 {"name":"two","path":"/Users/me/dotfiles-client","profiles":["client-acme"]}]

Configured path

The literal value from .cubby.toml, like ../src or ~/dotfiles.

Resolved path

The cleaned source root Cubby actually uses after expanding ~ and resolving relative paths from the host root. This is the path shown in table and JSON output.

Feature 2

Source names become strict

Required

Every [[source]] must have a real name.

Unique

Duplicate names are rejected now, before --source selection arrives in Milestone 7.

Restricted

Names must match ^[A-Za-z0-9_-]+$: letters, digits, dashes, and underscores only.

Think of source names as stable IDs. No spaces, dots, slashes, or decorative punctuation.

Feature 3

Missing profiles become diagnostics

If you select work and only one source provides it, the other sources are skipped with a clear note:

source "personal" does not declare selected profile "work"; skipping

Still an error

A selected profile declared by no registered source fails before link/unlink side effects.

Only a diagnostic

A selected profile missing from one source is okay if another source declares it.

Mental model

There are three profile sets

1

Host default profiles

Top-level profiles in .cubby.toml. These are defaults only, not an allowlist.

2

Effective selected profiles

What this invocation actually selected after precedence: host defaults < CUBBY_PROFILE < --profile.

3

Per-source declared profiles

Each source's cubby.toml declares what that source can provide.

Core logic

How link/unlink should think

selected = effective profiles for this command

validate selected is not empty
validate every selected profile exists in at least one source

for each source in registration order:
  missing = selected - source.declared_profiles
  emit stderr diagnostics for each missing profile

  eligible = selected ∩ source.declared_profiles
  discover profile files for eligible
  add those files to the link/unlink plan

The important idea: global validation protects the whole command; per-source intersection decides actual work.

Core logic

How source list should think

project = LoadProject() // strict
items = collect name, resolved path, profiles from project.Sources

if --json:
  json.NewEncoder(stdout).Encode(items)
else:
  render Lipgloss table with NAME, PATH, PROFILES

Registration order stays stable in both modes. The default table is for humans; --json is for scripts and should contain no styling.

Safety

Collisions stay governed by Milestone 4

Default mode

Cross-source collisions are fatal. No links are created when the plan has fatal conflicts.

--ignore-conflicts

The first registered source wins. Later colliding paths are skipped and reported, never overwritten.

Milestone 5 does not weaken file safety. It only proves that safety across multiple sources.

Inventory union

Profile list and gitignore stay source-derived

Both commands should ignore host defaults and use the union of what sources declare.

profile list

Sorted union of all source-declared profiles:

client-acme
personal
work

gitignore check/sync

Required patterns for every source-declared profile:

*.work.*
*.work
*.client-acme.*
*.client-acme

Implementation order

The plan in five slices

1

Config validation: require explicit unique source names matching ^[A-Za-z0-9_-]+$.

2

Source list: strict loader, Lipgloss table output, resolved paths, profiles CSV, and --json.

3

Diagnostics: stderr notes for source/profile misses in link, unlink, and dry-run.

4

Multi-source e2e: prove link/unlink, explicit selection, defaults, and collisions.

5

Inventory/gitignore e2e: prove source list, profile union, and gitignore union.

What you get after

A stronger multi-source foundation

After Milestone 5, Cubby can answer three practical questions:

What sources are registered? What profiles exist? What work applies to each source?

That foundation sets up Milestone 6 state discovery and Milestone 7 source-scoped commands without changing the core mental model.

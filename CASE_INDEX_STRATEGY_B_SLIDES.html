Terminology

What “source-source” means

A source-source collision is a conflict between two files Cubby discovered from configured source repos before looking at the host filesystem.

source-source

Two planned source files collide

source one has: foo.work
source two has: FOO.work

// or inside one source:
src/foo.work
src/FOO.work

Cubby already knows both strings from source discovery, so it can compare folded planned paths directly.

source-host

A planned file collides with existing host state

source projects: FOO.work
host already has: foo.work

This needs the host case index because Cubby must learn the host’s actual spelling from directory entries.

Short version: source-source means “planned file vs planned file”; source-host means “planned file vs something already in the host repo.”

Slide 6

Why “bounded” matters

whole host walk

Walk everything

Read every directory and file in the host repo. Simple conceptually, but can be slow and can hit unrelated permissions/errors.

bounded index

Walk only what the plan can touch

If the plan contains nvim/init.work.lua, we inspect host root and any matching/needed parent directories. We do not walk unrelated directories.

host/ ├── .git/ not relevant ├── huge-vendor-tree/ not relevant ├── Nvim/ relevant because plan has nvim/... │ └── old.lua └── foo.work relevant because plan has FOO.work
Slide 7

The build algorithm: high level

  1. Start with all planned relpaths.
  2. For each planned relpath, split into components.
  3. Read host root directory entries and index their actual spellings.
  4. For each component prefix, find host entries whose folded name matches the planned component.
  5. If a matching entry is a directory, continue into it to index the next level.
  6. Stop when no matching parent exists; missing parents mean there is no deeper existing host conflict.
planned "nvim/init.work.lua"
components: ["nvim", "init.work.lua"]
read host/
  found "Nvim" folds to "nvim" → index "nvim" as actual "Nvim"
  because Nvim is dir, read host/Nvim/
    found possible children...
Slide 8

Build pseudocode

func BuildHostCaseIndex(hostRoot string, plannedRelPaths []string) (*HostCaseIndex, error) {
  idx := &HostCaseIndex{Entries: map[string][]HostEntry{}}
  readCache := map[string][]DirEntry{} // abs dir → sorted entries

  for _, rel := range plannedRelPaths {
    rel = filepath.Clean(rel)
    components := split(rel)

    // active parents are actual host dirs that match the planned prefix so far.
    active := []actualPrefix{{AbsDir: hostRoot, RelPrefix: ""}}

    for _, wantName := range components {
      next := []actualPrefix{}
      wantFold := foldSegment(wantName)

      for _, parent := range active {
        entries, err := readSortedDirCached(readCache, parent.AbsDir)
        if missing(parent.AbsDir) { continue }
        if err != nil { return nil, err }

        for _, entry := range entries {
          if foldSegment(entry.Name) != wantFold { continue }

          actualRel := joinRel(parent.RelPrefix, entry.Name)
          idx.add(actualRel, entry.IsDir)

          if entry.IsDir {
            next = append(next, actualPrefix{
              AbsDir: filepath.Join(parent.AbsDir, entry.Name),
              RelPrefix: actualRel,
            })
          }
        }
      }

      if len(next) == 0 {
        break // no existing matching parent; deeper host path cannot exist
      }
      active = next
    }
  }
  return idx, nil
}

This is intentionally shape-level pseudocode. Exact Go code should use stable error wrapping and filepath-safe helpers.

Slide 9

Check algorithm

For a planned relpath, ask: does any existing host path prefix fold to my prefix but use different casing?

func (idx *HostCaseIndex) CaseVariantPrefix(plannedRel string) (actualRel string, ok bool) {
  plannedRel = filepath.Clean(plannedRel)
  prefixes := prefixesOf(plannedRel)
  // "nvim/init.work.lua" → ["nvim", "nvim/init.work.lua"]

  for _, prefix := range prefixes {
    folded := foldKey(prefix)
    for _, existing := range idx.Entries[folded] {
      if existing.RelPath != prefix {
        return existing.RelPath, true
      }
    }
  }
  return "", false
}
Run this check before treating an exact path as NOOP. Otherwise a correctly resolving symlink with wrong casing can hide a case-policy conflict.

Slide 20

Where Strategy B fits in code

func PlanLink(hostRoot string, sources []SourceFiles, opts PlanOptions) (Plan, error) {
  planned := collectBaseActions(hostRoot, sources)

  actions, survivors := classifyPlannedCollisions(planned, opts)

  var hostIndex *HostCaseIndex
  if !opts.CaseSensitive {
    hostIndex = BuildHostCaseIndex(hostRoot, relPaths(survivors))
  }

  for _, action := range survivors {
    if !opts.CaseSensitive {
      if actual, ok := hostIndex.CaseVariantPrefix(action.RelPath); ok {
        actions = append(actions, conflictOrSkip(action, "host path case conflict with " + actual, opts))
        continue
      }
    }
    actions = append(actions, classifyExact(action, opts))
  }
  return Plan{Actions: actions}, nil
}

The separation matters: planned collisions are pure; host case policy is index lookup; exact filesystem classification remains unchanged.

Slide 22

Where it can fall short

Unicode

Lowercase is not full Unicode case folding

strings.ToLower is the planned initial behavior. Full Unicode case folding and filesystem Unicode normalization are explicitly out of scope for this milestone.

TOCTOU

Filesystem races remain possible

The host can change after planning. Apply must still use safe create semantics and return errors rather than overwrite.

No directory symlinks

Directory symlinks are out of scope

Cubby links regular source files only. It should not project source directories or source symlinks, and the host case index should not traverse symlinked directories as managed targets.

Case-sensitive opt-in

case_sensitive = true disables policy checks

The OS may still reject creates on a case-insensitive filesystem. That is an apply-time filesystem limitation, not a Cubby policy conflict.

Slide 23

Implementation pattern

1

Collect

Gather planned relpaths. Read only host directories whose names fold-match planned prefixes.

2

Index

Store actual spellings under folded keys. Sort entries for deterministic output.

3

Check

For each planned prefix, compare planned spelling to indexed actual spelling. Conflict before exact classification.

This isolates filesystem discovery from policy decisions. Most unit tests can target the index and checker without involving symlink creation.

Slide 25

Final takeaway

The complexity is not in comparing strings. The complexity is in safely learning the real strings that already exist on disk, across filesystems with different case behavior, before any mutation happens.

Filesystem reality
actual host spelling
→
Bounded host case index
folded key → actual relpath
→
Simple policy
compare strings, conflict before apply
Strategy B is the bridge: it pays a small, explicit collection cost so the planner can make deterministic, testable, string-based decisions.

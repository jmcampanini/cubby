1. Doctor can miss invalid source configs

The diagnostic loader accepts a source config after loading TOML and checking that profiles exist, but it does not validate fields that are only exercised later, especially source ignore globs.

P2 internal/config/diagnostic.go
What it is

LoadProjectDiagnosticsWithHostConfig is intentionally best-effort so doctor can report multiple problems. However, it currently marks a source as valid even when the source's cubby.toml contains an invalid ignore pattern such as ignore = ["["].

The strict link path later calls profile discovery, which compiles those ignore rules and fails. Doctor may never reach that code path.

How it comes up

A host has a registered source with declared profiles and an invalid ignore glob. If the host has no requested profiles, no managed links that hit ignore checking, and the required gitignore patterns are present, doctor can say everything is healthy.

# src/cubby.toml
profiles = ["work"]
ignore = ["["]

# host/.cubby.toml
[[source]]
name = "src"
path = "../src"

$ cubby doctor
# exits 0, no output

$ cubby link --profile work
# fails: invalid ignore pattern "["
Why this is a problem
  • doctor is supposed to be the command users trust before changing host state. A false green result undermines that role.
  • The source is not truly valid; later commands can fail before creating links, even though health checks passed.
  • The failure is environment-dependent: it may appear only when a profile is selected later, which makes the issue harder to connect back to the source config.
Plain English: Doctor is checking whether the house is safe, but it never tests one of the locks. Later, when you actually try to use that lock, the command fails.

Fix options

Shared source-config validation Recommended

Add a shared validation helper for loaded source configs and call it from both strict loading and diagnostic loading. That helper should validate profiles and compile/validate ignore patterns.

Pros

  • Prevents strict and diagnostic loading from drifting.
  • Reports the source as an invalid_config issue in doctor.
  • Catches the problem without walking source files.

Cons

  • Slight refactor across config/profilefiles boundaries.
  • May require exposing a small ValidateIgnore helper.

Validate ignore rules only in diagnostic loading

After reading sourceCfg in loadRegisteredSourceDiagnostic, compile the ignore rules and convert failures into a SourceIssue.

Pros

  • Small, targeted patch.
  • Directly fixes the false-green doctor case.

Cons

  • Leaves validation rules split across code paths.
  • Future source-config fields could be missed again.

Make doctor discover all source profile files

Force doctor to run profile discovery for all declared profiles, even when no profiles are requested, so ignore rules are compiled as a side effect.

Pros

  • Uses existing discovery behavior.
  • Can catch more file-discovery errors.

Cons

  • Indirect: config validation happens as a side effect.
  • More filesystem work for doctor.
  • Less clear error reporting than a structured source issue.
Recommendation: use shared source-config validation. It is the most maintainable fix because it makes "valid source config" mean the same thing everywhere: link, status, prune, and doctor all agree on what is usable.

2. ignore_conflicts hides doctor conflict diagnostics

doctor passes the host's ignore_conflicts setting into link planning, then only reports fatal conflicts. When conflict ignoring is enabled, real conflicts become skips and disappear from doctor output.

P2 cmd/doctor.go
What it is

The link planner has two jobs: find conflicts and apply the host policy for whether conflicts are fatal or skipped. Doctor currently asks the planner to honor the host policy, then looks only at FatalConflicts().

That means ignore_conflicts = true changes doctor from "report conflicts" to "pretend conflicts are not health issues."

How it comes up

A requested profile has a source file that should project to foo.work, but the host already has a regular file at foo.work. Link would skip it because conflicts are ignored. Doctor exits successfully.

# host/.cubby.toml
profiles = ["work"]
ignore_conflicts = true

[[source]]
name = "src"
path = "../src"

# host/foo.work already exists as a regular file
# src/foo.work also exists

$ cubby doctor
# exits 0

$ cubby link
# skips foo.work; requested profile is not fully linked
Why this is a problem
  • A skipped link is still an incomplete requested profile. Doctor should surface that state, even if normal linking is configured to continue.
  • Users can believe the repo is healthy while some profile files are silently not projected into the host.
  • The Milestone 6 goal says doctor should include conflict diagnostics; honoring ignore_conflicts here defeats that check.
Plain English: ignore_conflicts is a safety policy for link: "do not stop the whole operation." Doctor should answer a different question: "is anything blocking the desired state?"

Fix options

Plan doctor conflicts with ignoring disabled Recommended

Call PlanLink from doctor with IgnoreConflicts: false, regardless of the host config, and report FatalConflicts().

Pros

  • Very small change.
  • Matches doctor semantics: healthy means requested profiles can be fully applied.
  • Keeps output simple: conflicts remain CONFLICT.

Cons

  • Doctor may report conflicts that link would intentionally skip.
  • Needs documentation clarity that doctor treats skipped requested files as health issues.

Report conflict skips as doctor issues

Keep IgnoreConflicts as configured, but also inspect plan.Skips() and report skips caused by host conflicts as CONFLICT ... ignored=true.

Pros

  • Reflects exactly what link would do.
  • Can explain that the conflict is configured to be skipped.

Cons

  • More complex: doctor must distinguish conflict skips from other skip types.
  • Output contract grows another nuance.

Split planning facts from policy

Refactor linkops so plans retain raw conflict facts separately from the configured action (skip or conflict).

Pros

  • Clean model: doctor can ask for facts, link can apply policy.
  • More extensible for future health checks.

Cons

  • Larger refactor than this bug requires.
  • More tests need updating around plan shape and rendering.
Recommendation: plan doctor conflicts with IgnoreConflicts: false. It is direct, easy to test, and preserves the meaning of doctor as an aggregate health check rather than a dry-run of the configured best-effort link behavior.

3. Registered source symlink roots are classified as drift

When a registered source path inside the host repo is itself a symlink to the real source directory, host-link discovery can classify that source-root symlink as a managed Cubby link with drift.

P3 internal/hostlinks/hostlinks.go
What it is

The walker skips registered source directories only in the d.IsDir() branch. But filepath.WalkDir sees a symlink-to-directory as a symlink entry, not as a directory to descend into.

So a configured source root like host/src -> ../real-src reaches symlink classification. Its target resolves inside the source root, so it is treated as a managed link. Because the host-relative path is src while the source-relative target is ., it is reported as drift.

How it comes up

This appears when someone keeps a convenient source pointer inside the host repo and registers that pointer as the source path.

host/
  .cubby.toml
  src -> ../real-src        # registered source path
real-src/
  cubby.toml
  foo.work

# host/.cubby.toml
[[source]]
name = "src"
path = "./src"

$ cubby status
DRIFT src [source=src target=. reasons=path mismatch,unknown profile]
Why this is a problem
  • The source-root symlink is configuration plumbing, not a Cubby-managed profile file.
  • status and doctor can report false drift in an otherwise healthy setup.
  • Users may chase a nonexistent problem or avoid a valid source-layout pattern.
Plain English: Cubby is mistaking the sign pointing to the source repo for one of the files it created.

Fix options

Skip registered source paths before symlink classification Recommended

Move the isRegisteredSourceDir(path, roots) check before the directory-only branch. If the path is a directory, return filepath.SkipDir; if it is a symlink/file entry at a registered source root, return nil.

Pros

  • Minimal, targeted change.
  • Matches the existing intent: source roots inside host are not host state.
  • Avoids changing source path resolution or link target behavior.

Cons

  • Ignores a source-root symlink even if a user expected it to be reported.
  • Requires careful handling because SkipDir is only appropriate for directories.

Filter source-root symlinks during classification

Inside classify, if the host path is exactly the registered source root and the target resolves to that root, return unmanaged.

Pros

  • Localized to classification.
  • Can be very explicit about the false-positive case.

Cons

  • Narrower special case than the walker-level rule.
  • Classification now needs extra awareness of source-root plumbing.

Canonicalize source roots to real paths

Resolve source roots with filepath.EvalSymlinks during project loading and use canonical paths throughout discovery.

Pros

  • Can simplify some ownership comparisons.
  • Reduces ambiguity between lexical and real source paths.

Cons

  • Potentially backwards-incompatible: link targets may change from the configured symlink path to the real path.
  • Does not by itself skip the host-side symlink entry unless walker logic also changes.
  • Too broad for this false positive.
Recommendation: skip registered source paths before symlink classification. It fixes the false positive where it is introduced, preserves existing path semantics, and aligns with the stated discovery rule that source roots inside the host should not be mistaken for host state.

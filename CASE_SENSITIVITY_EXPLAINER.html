Why case sensitivity is not “just lower-case both strings”

For planned source files, it mostly is string comparison. The hard part is comparing a desired future path against what already exists on disk, because the filesystem may or may not treat different casing as the same path, and it usually does not hand us the real spelling unless we ask the parent directory.

The desired safety rule

By default, Cubby behaves case-insitively for projected host paths. If two paths would be the same after case folding, Cubby must report a conflict/skip before creating anything.

Existing exact path

The host already has exactly foo.work. We can Lstat(host/foo.work) and classify regular file, directory, symlink, etc.

filesystem + strings

Existing different-cased path

The source projects FOO.work, but host has foo.work. To know that, we must inspect directory entries and compare their names.

The mental model: strings after we have names

The simple model is correct once we have two concrete path strings:

collisionKey = filepath.Clean(relPath)
if !caseSensitive {
  collisionKey = strings.ToLower(collisionKey)
}

The problem is that for existing host paths, we often only have one string: the path Cubby wants to create. We do not necessarily know the actual spelling of any existing path until we read directory entries.

Why the filesystem makes this tricky

ScenarioWhat Lstat("FOO.work") doesWhat Cubby needs
Case-sensitive filesystem, host has foo.work Returns “not exists” for FOO.work. Still detect foo.work by reading the parent directory and folding names.
Case-insensitive filesystem, host has foo.work May succeed for FOO.work, because the OS aliases the spelling. Still report a case-policy conflict if actual spelling differs. Exact Lstat success is not enough.
Nested parent variant: host has Nvim/, source projects nvim/init.work.lua On case-sensitive FS, host/nvim/init.work.lua is missing. Conflict with existing parent variant Nvim; otherwise Cubby creates a second sibling directory nvim.
Correct symlink but wrong casing The symlink may resolve correctly. Case policy should run before “already linked” success, or casing drift is hidden.

Current implementation strategy

The current code has two separate checks:

  1. Planned source-source collision: store each planned relpath in seen[collisionKey(relPath)].
  2. Existing host case variant: after exact Lstat says the path is missing, walk directory entries looking for case-fold matches.

The two recurring bug shapes

bug shape 1

Exact lookup succeeds, but spelling differs

Host directory contains: foo.work
Source projects:         FOO.work
Filesystem:              case-insensitive

Lstat("FOO.work") can succeed, so the current “only scan on IsNotExist” rule never asks “what is the real spelling in the directory?”

bug shape 2

Parent differs, leaf does not exist

Host directory contains: Nvim/
Source projects:         nvim/init.work.lua
Filesystem:              case-sensitive

The current traversal can stop at “no full leaf match” and return no conflict, even though creating nvim/ violates the host-wide case-insensitive policy.

Decisions that make this not straightforward

DecisionWhy it adds complexity
Default policy is case-insensitive everywhere Linux may allow foo and FOO, but Cubby still rejects them by default for cross-platform safety.
Plan before mutation We must find every conflict before creating any symlinks, so the detection has to be complete during planning.
Existing host files are user data We cannot “try creating and see what happens”; that risks partial mutation or OS-specific behavior.
Nested paths count component-by-component A conflict may be with a parent directory spelling, not only the final file name.
Correct symlink detection is separate from case policy A symlink can resolve to the right source but still live at the wrong-cased projected path.

Alternative strategies

All strategies still end in string comparison. The choice is about how we obtain and organize the existing host path strings.

Strategy A

Patch current approach: component lookup for every planned path

Before exact classification, walk from host root through each relpath component. At every parent, read entries, find same-folded names, and record whether the actual spelling differs. Return a conflict as soon as a differing parent exists, even if the leaf is missing.

lookupCase(hostRoot, "nvim/init.work.lua")
  sees "Nvim" matching "nvim" with different case
  => conflict with "Nvim" before considering the leaf

Pros: smallest change, bounded to planned paths. Cons: still interleaves filesystem traversal and classification, which is where subtle bugs have been appearing.

Strategy B recommended

Build a bounded host case index, then compare strings

For all planned paths, collect only the relevant host directory entries along their ancestor chains. Store them in a folded map/trie. Then planning becomes mostly pure string comparison against that index.

hostIndex["nvim"] = "Nvim"
hostIndex["foo.work"] = "foo.work"

planned "nvim/init.work.lua" => folded prefix "nvim" exists as "Nvim" => conflict
planned "FOO.work" => folded full path exists as "foo.work" => conflict

Pros: makes the naive model true after one explicit indexing step; easier to unit test; stable decisions. Cons: more upfront code and a small cache structure.

Strategy C

Walk the whole host repo into a case-folded path set

Traverse the host root once, build foldedRelPath -> actualRelPath for every existing path, then compare all planned paths to that set.

Pros: conceptually simplest: everything is strings. Cons: can be expensive for large host repos, may trip over permissions, and must decide what to skip (.git, ignored dirs, etc.).

Recommended direction

I would move away from the current on-demand “scan only after missing” approach and use Strategy B: a bounded host case index. It preserves the simple string-comparison model while making the filesystem interaction explicit and testable.

Planner order with Strategy B

  1. Discover selected source files.
  2. Build planned collision keys from relpath strings; detect source-source case collisions.
  3. Build a bounded host case index for planned relpaths and their parent chains.
  4. For each non-colliding planned item, ask the index: “does any existing host path or parent path fold to my path but differ in spelling?”
  5. If yes, emit CONFLICT/SKIP. If no, run exact Lstat classification for create/noop/exact conflict.
  6. If any fatal conflict exists, apply nothing.

Test checklist that should prevent repeat bugs

Pure planning

  • foo.work + FOO.work in same source conflicts by default.
  • Same pair across two sources conflicts by default.
  • case_sensitive = true disables policy collisions.

Existing host state

  • Existing leaf variant: host foo.work, planned FOO.work.
  • Existing parent variant with missing leaf: host Nvim/, planned nvim/init.work.lua.
  • Exact path still uses exact classification: regular file, directory, expected symlink, unexpected symlink.
Important: tests should not only run on one filesystem behavior. Unit tests for the case index can avoid relying on OS case sensitivity by feeding synthetic directory entries or temporary trees with guarded expectations.
